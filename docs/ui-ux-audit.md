# BookJumpr — UI/UX Audit

_Written as part of the pre-launch polish pass. Organized by severity/effort: fixed now, safe-but-deferred, and judgment calls that need your sign-off._

## Fixed in this pass

- **Cover-wall alignment bug** (the reported issue): in the book-detail "Mentions"/"Mentioned by" grids and the author-page grid, title/author text sat *outside* the cover graphic with no line-clamp and no reserved height, so a wrapping title or author name pushed that card's "cited by" line down relative to its row-mates. Fixed at all three locations in `index.html` with `align-items:start` on the grid, a 2-line clamp + reserved min-height on the title, and single-line ellipsis truncation on author/"cited by" text — so every card in a row now occupies identical vertical space regardless of text length.
- **`--accent` design token was a no-op**: defined as `#141414` (same as `--ink`) in the raw CSS, silently overwritten to the real `#9c3d22` by JS after mount. This caused a brief flash-of-incorrect-color on load and meant the static CSS source was misleading about the actual brand color. Now `--accent` is correct at parse time.
- **`--mut` token added** and applied to the 11 places using the identical hardcoded `opacity:.55` value for de-emphasized text (section labels like "Book"/"Author"/"Stats", and several captions in the network view's own CSS) — consolidating the single most redundant of the 9 different opacity values found in an ad hoc sweep across the codebase. The other values (.6/.65/.7/.75/.8) were left alone; a full type/opacity-scale rebuild is a larger design decision, not a mechanical fix.
- **Focus ring restored on inputs**: `input:focus{outline:none;}` previously removed all keyboard-focus indication site-wide with nothing put back — a real accessibility regression, since sighted keyboard users (and anyone using switch/voice control) had no way to see which input was focused. A visible `outline` in the accent color now appears on `:focus-visible` (keyboard focus), without interfering with mouse-click focus or the inputs' existing `box-shadow` styling.
- **`aria-label`s added** to every icon-only button that previously relied solely on a `title` tooltip: the header's mobile icon nav (Network/Stats/About), the home hero's icon row (same three), and the network view's filter/zoom-in/zoom-out/fit buttons.
- **Active-route indication added to navigation**: the desktop text nav (Network/Stats/About) now underlines and colors the current section in accent; the mobile icon nav shows full opacity + an ink border on the active icon. Previously `pgNetwork`/`pgStats`/`pgAbout` were computed in `app.js` but never applied to the nav's own styling — there was no way to tell which section you were in without reading the page content.
- **Constellation view is now lazy-loaded** (`constellation-data.js` + `constellation-view.js`, ~1.2MB combined) only on first visit to `#/network`, instead of on every single page view. This is a UX win as much as a performance one — every other route now loads meaningfully faster.

## Safe but deferred (not done this pass — logged for a future cleanup)

- **Remaining opacity values** (.6/.65/.7/.75/.8) are still hardcoded ad hoc across `index.html` rather than tokenized. Consolidating them is more invasive than the single `.55` sweep done here and touches many more call sites for a purely internal-consistency benefit.
- **Cover-wall grid DRY extraction**: the three fixed grids (book Mentions, book Mentioned-by, author page) are copy-pasted markup, not a shared partial — the template engine (`app.js`'s `{{ }}`/`<sc-for>` compiler) has no include mechanism. A cleaner long-term fix would extract a `P.bookCardGrid(list)` helper mirroring the existing `{{ miniGraph }}` pattern (a fully JS-rendered subtree assigned to one template value, `app.js`'s `miniGraph()` method). Left alone here since the CSS-only fix already fully resolves the reported bug with zero risk to the render pipeline; the refactor is a maintainability nice-to-have, not a user-visible improvement.
- **Icon-only `.x` close buttons inside `constellation-view.js`** (the chip and detail-panel close buttons, 4 total) still have no `aria-label`. These are managed by the constellation engine's own DOM code (not the `app.js` template), and weren't in this pass's agreed scope — flagging for a follow-up.
- **Three independent "is this mobile" mechanisms** coexist: `app.js`'s three JS breakpoints (`vw<640`, `vw<900`, `vw<1100`, used for different layout decisions), `constellation-view.js`'s own `clientWidth<640` check, and `admin.html`'s separate CSS media queries. None of these are wrong per se, but they're undocumented and not unified — a future pass could consolidate to one signal (e.g. a single `--mobile` custom property or a shared JS breakpoint constant) so "is this mobile" means the same threshold everywhere.
- **404 / bad hash route handling**: an unrecognized hash like `#/asdkjfh/asdkjfh` currently falls through to the author-page empty state ("No books on file for this author yet."). This is a reasonable fallback, not a crash, but it's worth a deliberate look at whether a clearer "nothing here" message would read better — a judgment call, not a code defect.

## Judgment calls — needs your design sign-off before touching

- **No spacing or type scale**: font sizes are declared ad hoc in 26+ places in `index.html` with no shared scale (e.g. `13.5px`, `10.5px`, `11px`, `12.5px` all appearing as one-off literal values). This works today because the values were chosen carefully by hand, but it means every future addition risks drifting from the existing rhythm. Building a real scale (a handful of named sizes covering the current range) is a genuine design decision, not a mechanical extraction — recommend doing this deliberately rather than as a side effect of this pass.
- **Minification / bundling pipeline**: `app.js` (70KB) and `constellation-view.js` (71KB) are unminified, comment-heavy hand-written source loaded directly with no build step. This is a legitimate choice for a project that's deliberately dependency-free and hackable — introducing a bundler changes the project's whole "no build step" character (see `README.md`: "It's a static site — no build step, no server-side code"). Worth a conscious decision, not something to bolt on quietly.
- **Full breakpoint unification** (see above) — touches layout logic across `app.js` and `constellation-view.js` simultaneously; safer as a dedicated pass with visual QA at each breakpoint than folded into a polish pass.

## What this means for launch

None of the deferred or judgment-call items are launch-blocking. The cover-wall bug (the concrete issue that prompted this audit) is fixed, along with several real accessibility regressions (missing focus indication, missing labels on icon-only controls) that are worth fixing before public launch regardless of the original ask. The remaining items are legitimate future design/engineering investments, not defects.
